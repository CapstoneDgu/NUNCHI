# 눈치 키오스크 — 10분 발표 계획서

> **제목**: 눈치 키오스크 — AI와 눈 인식 기반 접근성 주문 시스템
> **핵심 메시지**: 기존 키오스크의 복잡한 조작 문제를 풀기 위해, **일반 주문 · AI 대화 주문 · 눈 인식 보조 입력 · 관리자/모니터링**을 하나의 흐름에 통합한 접근성 중심 키오스크를 **설계하고 구현했다.**
> **구성 원칙**: 데모는 일부일 뿐. "왜 만들었고(문제) → 어떻게 설계했고(설계) → 실제로 돌아가고(구현) → 검증했고(테스트) → 사회적 의미(영향·윤리)"를 **짧고 강하게.** 데모는 **3분 20초**로 제한.

## 2. 슬라이드 구성 (10장) — 자료 + 핵심 3줄

### [1] 제목
- **자료**: "눈치 키오스크 / AI와 눈 인식 기반 접근성 주문 시스템", 팀명·학교, 키오스크 비주얼 1컷
- **핵심 3줄**
  1. AI와 눈 인식 기반 접근성 키오스크 "눈치".
  2. 누구도 배제하지 않는 주문을 한 줄 슬로건으로.
  3. 첫 장부터 "접근성 + AI"라는 정체성을 각인.

### [2] 문제정의 (0:00~0:50)
- **자료**: 복잡한 키오스크 화면, 휠체어·고령층·손 사용 불편 상황 이미지
- **핵심 3줄**
  1. 키오스크는 빠르고 저렴하지만 모두에게 편하지는 않다.
  2. 디지털 취약계층·휠체어·손 사용 불편 사용자는 탐색과 터치 자체가 장벽.
  3. → "예쁜 화면"이 아니라 **입력·탐색·주문·운영이 연결된 종합 공학 문제**로 정의.

### [3] 해결 아이디어 / 차별점 (0:50~1:40)
- **자료**: 4개 기능 아이콘 — ①일반 주문 ②AI 대화 주문 ③눈 인식 보조 입력 ④관리자/모니터링
- **핵심 3줄**
  1. 네 기능으로 푼다: 일반 주문 / AI 대화 주문 / 눈 인식 보조 입력 / 관리자·모니터링.
  2. 차별점은 단일 기능이 아니라 **접근성·AI 추천·운영 안정성을 한 흐름에 통합**한 것.
  3. 그리고 "눈치" — **체류·반복·표정으로 망설임을 먼저 읽어** 추천한다(프로젝트명의 근거).

### [4] 전체 시스템 구조 (1:40~2:40) ★도구사용 10점
- **자료**: 아키텍처 그림
  ```
  사용자 키오스크 화면(정적 HTML/CSS/JS)
        ↓
  Spring Boot 백엔드  ──  PostgreSQL / Redis
        ↓
  FastAPI AI 서버  ──  LangGraph / MCP Tool
        ↓
  OpenCV + MediaPipe 눈 인식 엔진(로컬)
        ↓
  Actuator → Prometheus → Grafana
  ```
- **핵심 3줄**
  1. 프론트(정적) ↔ Spring Boot(주문·메뉴·관리자·AI프록시·비전 이벤트 API) ↔ FastAPI(LangGraph·MCP).
  2. 눈 인식은 웹캠 → OpenCV/MediaPipe 로컬 비전 엔진 → WebSocket으로 YES/NO 이벤트 전달.
  3. 운영은 Actuator+Prometheus+Grafana로 요청수·응답시간·5xx 에러율 관측.

### [5] 주요 구현 기능 (2:40~3:30) ★구현 완성도
- **자료**: 기능 목록(사용자/관리자/운영 3분류)
- **핵심 3줄**
  1. 사용자: 방식·매장/포장 선택, 메뉴 조회, 장바구니, 주문 확정, AI 추천, 눈 인식 YES/NO.
  2. 관리자: 주문 내역, 메뉴 관리, 품절/추천 관리.
  3. 운영: Actuator 상태, Prometheus 수집, Grafana 시각화 — "다 동작 중".

### [6] 데모 1 — 일반 주문 (3:30~4:20)
- **자료**: 캡처 3~4컷(시작→일반선택→매장/포장→메뉴→상세→담기)
- **핵심 3줄**
  1. 방식 선택 → 매장/포장 → 카테고리 메뉴(가격·이미지·추천·품절) → 수량 → 담기.
  2. 이 기본 흐름이 AI·눈 인식 주문에서도 동일하게 재사용된다.
  3. 터치에 익숙한 사용자를 위한 빠르고 직접적인 경로.

### [7] 데모 2 — AI 대화 주문 (4:20~5:20)
- **자료**: "인기 메뉴 추천해줘" → 결과 → "맵지 않은 메뉴" → 재추천 → 담기
- **핵심 3줄**
  1. 자연어 요청 → AI가 **실제 DB 메뉴 데이터** 기반으로 추천(문장만 생성 아님).
  2. 조건을 바꾸면 즉시 재추천, 추천 결과는 바로 주문 흐름으로 연결.
  3. 메뉴를 모르는 첫 방문자의 탐색 부담을 없앤다 — 여기서 "눈치"가 망설임을 먼저 살핀다.

### [8] 데모 3 — 눈 인식 보조 입력 (5:20~6:20) ★차별화
- **자료**: 비전 엔진 화면 — 오른쪽 응시=candidate YES, 더블 깜빡임=confirmed, 왼쪽 응시=NO
- **핵심 3줄**
  1. OpenCV/MediaPipe로 시선·깜빡임 인식: 우=YES 후보, 좌=NO 후보, **더블 깜빡임=확정**.
  2. 전체 조작이 아니라 **2지선다(매장/포장·담기·확정/취소)** 같은 정확도 중요 지점에만 적용.
  3. 눈 인식은 조작 대체가 아닌 **손 사용이 어려운 사용자를 위한 접근성 보조 입력**.

### [8-1] 데모 4 — 관리자/주문 확인 (6:20~6:50, 짧게)
- **자료**: 주문 확정 → 관리자 페이지 주문 확인 → 메뉴 관리 짧게
- **핵심 3줄**
  1. 주문 완료 시 서버·DB에 저장, 관리자 페이지에서 즉시 확인.
  2. 메뉴 정보·품절·추천 여부 관리 가능.
  3. 사용자 화면과 운영 화면이 **하나의 데이터 흐름**으로 연결.

### [9] 테스트 / 모니터링 (6:50~7:40) ★평가 방법
- **자료**: Grafana 대시보드, 기능 테스트 결과 표, AI 병목 분석
- **핵심 3줄**
  1. 일반/AI/눈 인식 각각 기능 흐름 기준으로 테스트(추천·확정·확정 여부).
  2. Actuator+Prometheus+Grafana로 응답시간·에러율 관측.
  3. **AI API 지연의 병목이 Spring/DB가 아니라 FastAPI 내부 Agent 실행 구간**임을 로그로 규명 — 구현에서 끝내지 않고 추적 체계까지 구성.

### [9-1] 개인별 역할 / 협업 (7:40~8:30) ★의사소통
- **자료**: 역할 분담 표, Trello 캡처
- **핵심 3줄**
  1. 기능 단위 분담 + Trello로 진행 관리(프론트·백엔드·AI·비전).
  2. AI는 FastAPI/LangGraph/MCP, 비전은 OpenCV/MediaPipe 별도 엔진+WebSocket 연동.
  3. (발표자) 서버·인프라·AI 연동·모니터링·비전 연동 구조 설계 담당.

### [10] 사회적 영향 / 윤리 + 한계·향후 (8:30~10:00) ★사회적영향·윤리·논리성
- **자료**: 접근성/개인정보/안정성 키워드, 한계·개선 목록, **향후 비전(MCP 개방) 한 줄**
- **핵심 3줄 (사회적 영향·윤리)**
  1. 고령층·휠체어·손 불편 사용자에게 더 다양한 주문 방식 제공(접근성).
  2. 윤리: **얼굴 영상은 서버 전송 없이 키오스크 로컬에서만 처리**, 눈 인식은 최종 확인 보조로만 제한.
  3. 운영자는 관리자·모니터링으로 서비스 안정 관리.
- **핵심 3줄 (한계·개선·비전)**
  1. 한계: 눈 인식은 조명·각도 영향 → 캘리브레이션·시선 smoothing·양안 평균으로 개선.
  2. AI 응답 지연 → 스트리밍·추천 캐싱·Agent 단계 최적화. 결제단말·프린터·장애대응 보강 필요.
  3. **향후 비전**: 엔진을 MCP 표준으로 감쌌기에 외부 LLM(예: Claude·동국대 AI팩) 확장 가능 — *20초 클립(배속 포함)으로만.*

## 6. 운영 팁

- **데모는 전체의 일부.** 데모 3:20, 나머지는 설계·검증·근거로 점수를 번다.
- 강조 우선순위: ①문제정의(왜) ②설계(어떻게 연결) ③구현(돌아가는가) ④검증(테스트·모니터링) ⑤사회적가치/윤리.
- 비전·Prometheus·Grafana·눈 인식·AI는 모두 **"이미 동작 중"** 으로 서술.
- AI 병목 분석([9])은 **공학적 깊이**를 보여주는 킬러 포인트 — 꼭 살린다.
- 윤리([10] 로컬 처리)는 5점이지만 **"카메라=프라이버시 우려"를 선제적으로 답**해 감점을 막는다.
