# A01 아바타 모드 — 프론트엔드 구현 계획

## Context (왜 이걸 만드는가)

NUNCHI 키오스크는 **일반 UI 모드**와 **아바타 UI 모드** 두 가지 주문 방식을 제공한다. 현재 일반 모드(N02~N04, P01~P06)와 진입 플로우(S00~S02)는 모두 구현 완료됐지만, **아바타 모드는 폴더만 비어있는 상태**(`flowA/`, `static/front/css/flowA/`, `static/front/js/flowA/` 모두 `.gitkeep` 만 존재).

S02-dine.js:11 에서 이미 `mode === "avatar"` 일 때 `/flowA/A01-avatar.html` 로 라우팅하도록 분기는 깔려있지만 실제 페이지가 없어 **현재는 404**. 이 구멍을 채워서 사용자가 "AI 대화" 모드를 선택하면 아바타 캐릭터(**크롱이**)와 대화하면서 주문할 수 있게 한다.

이번 PR의 스코프는 **UI/UX 마크업 + Mock 대화**까지. 실 음성 파이프라인(WebSocket/STT/TTS)·MCP UI 자동조작·FastAPI 연동은 이번 PR에서 제외하고 별도 이슈로 분리. 단, **"진짜 음성 대화처럼 보이고 멋있게"** 라는 사용자 요구를 살리기 위해 시각적 인터랙션(아바타 비디오 전환, 음성 파형, 타이핑 효과, Web Speech API 시도)은 적극 구현.

## 화면 구조 (720×1280, 위→아래)

> **비율 갱신**: 아바타 스테이지를 화면의 절반 이상으로 크게 키워 "크롱이"의 시각적 임팩트를 살리고,
> 대화 로그/미니카트/입력바는 콤팩트하게 정돈. 합계 1280px 정확히 맞춤.

```text
┌────────────────────────────────────────────┐
│ ① 상단바 (56px)                             │  ← 60→56

│                                            │
│ ② 아바타 스테이지 (720px ≈ 56% 화면)         │  ← 520→720, ★메인 임팩트
│   - <video src="통장님_대기.mp4">  (loop)   │

│                                            │

│ ③ 기승전결 인디케이터 (44px)                 │  ← 50→44
```

## 핵심 인터랙션 디자인 ("실제 대화처럼 멋지게")

### 아바타 비디오 cross-fade
- 두 `<video>` 엘리먼트를 겹쳐 배치 (`position:absolute`, opacity로 전환)
- `state.avatarMode === "idle"` → 대기.mp4 보임 / "talking" → 대화중.mp4 보임
- transition: opacity 350ms ease-out
- 전환 트리거: AI 발화 시작 시 talking, 발화 종료 시 idle

### AI 발화 — typewriter + 파형
- 말풍선 텍스트를 `for ... await setTimeout` 으로 한 글자씩 추가 (40~60ms/글자, 문장부호 시 200ms pause)
- 동시에 SVG 파형(5~7개 막대) CSS animation으로 위아래 출렁
- 같은 텍스트가 ④ 대화 로그에도 추가됨 (말풍선은 최신만, 로그는 누적)

### 사용자 발화 — 마이크 인터랙션
- 우하단 64px 큰 마이크 버튼(`.btn-mic`)
- 누르는 순간:
  1. 버튼 주위 ripple ring 3겹 (CSS `@keyframes ripple` , primary-300/200/100)
  2. 입력바 위에 "듣고 있어요..." 라이브 캡션
  3. **Web Speech API 지원 브라우저면 실제 STT 실행** (`SpeechRecognition`, `lang: 'ko-KR'`, `interimResults: true`)
  4. 미지원/거부 시 텍스트 입력 폴백 + Toast 안내
- 인식된 임시 텍스트가 입력창에 실시간 표시되다가 final → 전송

### 빠른응답 칩 (Quick Replies)
- FSM 단계마다 다른 칩 세트:
  - 기: 없음 (인사 → 자동 진행)
  - 승: `[추천해줘]` `[매콤한거]` `[가벼운거]` `[메뉴 보여줘]`
  - 전: `[더 담을게]` `[충분해]`
  - 결: `[결제할래]` `[수정할게]`
- 칩 클릭 = 사용자 발화로 처리 (텍스트가 사용자 버블에 뜨고 다음 흐름 진행)

### 기승전결 인디케이터
- 4점 dot + 연결선
- 현재 단계: primary-500 채움 + 살짝 scale(1.2) + 글로우
- 완료 단계: primary-300
- 미완료: neutral-200
- 단계 변경 시 dot에 bounce 애니메이션 (--ease-bounce, 350ms)

## FSM 설계

```js
const FSM = {
  states: ['opening', 'recommend', 'addmore', 'confirm'],
  current: 'opening',
  transitions: {
    opening:    { onDineConfirmed: 'recommend' },
    recommend:  { onCartHasItem:   'addmore'   },
    addmore:    { onUserDone:      'confirm'   },
    confirm:    { onUserAgree:     '__navigate_P01__' },
  },
};
```

- 각 state마다 **진입 시 자동 발화 스크립트**(mock):
  - `opening`: "안녕하세요! 저는 크롱이에요. 매장에서 드시나요, 포장하시나요?" (sessionStorage.dineOption 이미 있으면 즉시 다음으로)
  - `recommend`: "오늘은 뭘 드시고 싶으세요? 추천해드릴까요?"
  - `addmore`: "○○ 담아드렸어요! 더 담으실 메뉴가 있을까요?"
  - `confirm`: "총 ₩X,XXX 원이에요. 결제하러 갈까요?"
- 사용자 발화 의도 분류는 단순 키워드 매칭(mock):
  - 추천/메뉴/뭐있어 → 메뉴 카드 노출
  - 메뉴명 직접 언급 → 해당 메뉴 카드 + 담기 제안
  - 더/추가/또 → 같은 단계 유지
  - 끝/그만/결제/충분 → 다음 단계
- **"명확한 주문 시 점프 가능"** 요구사항: 사용자가 "○○ 두 개 담고 결제할래" 같은 발화 시 recommend → confirm 점프 (간이 의도 파서)

- **접근성**: 마이크 버튼 80px+, 텍스트 입력 18px+, focus-visible 유지

### Phase 2 — STT / TTS 실 연동 (음성 파이프라인) ★ 최우선

- 마이크 권한 요청 → `MediaRecorder` 로 PCM 캡처 → WebSocket 으로 FastAPI STT 서버에 청크 스트리밍
- 서버 응답: `interim` / `final` 텍스트 + confidence 점수
- A01-avatar.js의 `initSpeech()` 자리에 어댑터 패턴으로 swap (Web Speech API → WebSocket 클라이언트)
- confidence < 0.7 → "혹시 ○○ 말씀하셨나요?" 재확인 대화 (요구사항 4-3)

**TTS (Text-to-Speech)**
- 현재 typewriter 효과는 그대로 살리되, 동시에 TTS 오디오 재생 (입을 비디오로 cross-fade 한 통장님_대화중.mp4 와 동기화)
- `aiSpeak(text)` 가 typewriter + TTS 오디오를 병행 시작, `audio.onended` + `typewriterDone` 둘 다 끝날 때 idle 복귀
- TTS 음원: FastAPI에서 mp3/opus 스트림 → `<audio>` 즉시 재생, 또는 base64 응답 → Blob URL

**WebSocket 프로토콜 (제안)**
```text
client → server: { type: 'audio_chunk', sessionId, seq, pcm: <bytes> }
server → client: { type: 'stt_partial', text, confidence }
                 { type: 'stt_final', text, confidence }
                 { type: 'intent', intent, entities }   // FastAPI LLM 처리 결과
                 { type: 'ai_text', text }              // AI 응답 텍스트 (typewriter용)
                 { type: 'ai_audio', url|base64 }       // TTS 오디오
```

**A01에서 바뀌는 부분 (최소 침습)**
- `initSpeech()`: Web Speech API 호출 → WebSocket 연결 + `MediaRecorder` 스트림 송신
- `aiSpeak(text)`: typewriter 단독 → typewriter + audio.play() 동시 + 종료 동기화
- 키워드 매칭 의도 분류 → 서버 `intent` 메시지 수신으로 대체 (handleUserIntent 입력 형태만 변경)
- FSM 자체와 UI는 그대로 (덕분에 swap 비용 작음)

**검증 항목**
