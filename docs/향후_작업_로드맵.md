# 향후 작업 로드맵 (2026-05-07 작성)

## 1. 목적/배경

`docs/API연결_진행현황.md` 의 작업(S 공통·A flow·P flow API 연결)이 백엔드 의존성(FastAPI CORS·`menu.floor` 마이그레이션) 해소 후 마무리되면, 다음 두 영역을 이어서 진행한다.

- **항목 A — STT/TTS 를 OpenAI 로 도입**: 현재 STT 는 브라우저 Web Speech API, TTS 는 사실상 typewriter 효과뿐. 음성 UX 의 정확도/자연스러움이 키오스크 차별점의 핵심이라 우선순위 1.
- **항목 B — 키오스크 모드 UI 점진적 개선**: 화면별로 디자인 보강 + 이미지 추가. 항목 A 가 끝난 뒤 사용자 피드백 받아가며 점진적 진행.

본 문서는 두 영역의 큰 그림을 한 곳에 정리한 **로드맵**. 항목 A 는 즉시 실행 가능하도록 상세 spec 을 포함하고, 항목 B 는 시작 시점에 별도 spec 으로 분리한다.

## 2. 항목 A — STT/TTS OpenAI 도입 (우선순위 1)

### 2-1. 결정 요약 (브레인스토밍 결과)

| 결정 사항 | 선택 | 근거 |
|---|---|---|
| **STT 모델** | OpenAI Whisper API | 한국어 정확도 1위, 분당 $0.006, OpenAI 키 1개로 TTS 와 통합 |
| **TTS 모델** | OpenAI TTS (`tts-1` 모델, `nova`/`shimmer` 등 음성) | 한국어 자연스러움 ⭐⭐⭐⭐, 월 ~3,000원 수준, 키 통합 |
| **API 키 보안 / 통합 위치** | Spring 백엔드 proxy 경유 | 키 클라이언트 노출 차단 + 같은 origin 이라 CORS 무관 + 가이드 §4("STT/TTS 는 프론트 담당")와 정합 (호출 트리거가 프론트, FastAPI 는 텍스트만) |

### 2-2. 아키텍처 다이어그램

```
[키오스크 마이크]
      ↓ (MediaRecorder 로 webm/opus 캡처)
[프론트 audio-recorder.js]
      ↓ POST /api/voice/transcribe (multipart/form-data, audio blob)
[Spring VoiceController]
      ↓ Whisper API 호출 (multipart, OPENAI_API_KEY)
[OpenAI Whisper]
      ↓ { text } 응답
[Spring] → [프론트] (ApiResponse<{ text }>)
      ↓ ConvEngine.onUserUtterance(text)
      ↓
[A01.handleUserIntent] → FastAPI /api/order/chat → reply
      ↓
[A01.aiSpeak(reply)] (기존: typewriter 만)
      ↓ POST /api/voice/synthesize { text, voice }
[Spring VoiceController]
      ↓ TTS API 호출 (OPENAI_API_KEY)
[OpenAI TTS]
      ↓ audio binary 응답 (mp3)
[Spring] → [프론트] (audio/mpeg)
      ↓ Audio 객체로 재생 + typewriter 동시 진행
[스피커 출력]
```

### 2-3. Spring 백엔드 변경

신규 도메인 `voice` 추가:

```
domain/voice
  ├── controller/VoiceController.java
  │     · POST /api/voice/transcribe — multipart audio file → text
  │     · POST /api/voice/synthesize — { text, voice? } → audio/mpeg
  ├── service/VoiceService.java
  │     · transcribe(MultipartFile) / synthesize(String text, String voice)
  ├── client/OpenAiClient.java
  │     · WebClient/RestClient 래퍼 — Whisper / TTS 엔드포인트별 메서드
  ├── dto/request/SynthesizeRequest.java (record)
  └── dto/response/TranscribeResponse.java (record, { text } 만 노출)
```

`application.yml` (또는 application-dev.yml) 에 추가:

```yaml
openai:
  api-key: ${OPENAI_API_KEY}
  whisper:
    model: whisper-1
    language: ko
  tts:
    model: tts-1
    voice: nova   # alloy | echo | fable | onyx | nova | shimmer
    format: mp3
```

`.env` / `.env.example` 에 추가:

```
OPENAI_API_KEY=sk-... (실제 키는 .env 만, .gitignore 됨)
```

오디오 파일 사이즈 제한: Whisper 는 25MB. 일반 발화(10초 이내) 는 1MB 미만. multipart 업로드 한도 `spring.servlet.multipart.max-file-size: 25MB` 명시.

### 2-4. 프론트엔드 변경

**제거**:
- `flowA/conversation-engine.js` 의 Web Speech API 의존 (`SpeechRecognition`, `webkitSpeechRecognition` 분기 전부 제거)

**신규 모듈**:
- `static/front/js/flowA/audio-recorder.js` (UMD 패턴)
  - `start()` — MediaRecorder 시작, 음성 캡처
  - `stop()` — 정지 + Blob 반환
  - VAD(Voice Activity Detection) 로 침묵 감지 → 자동 stop (가이드의 침묵 3초 정책 유지)
  - 권한 거부/마이크 미연결 시 fallback 토스트

**변경**:
- `flowA/conversation-engine.js`
  - 청취 시작/종료를 audio-recorder 호출로 교체
  - final 결과는 Spring `/api/voice/transcribe` 응답으로 받음
  - 상태 머신(INACTIVE/AI_SPEAKING/LISTENING/THINKING) 유지 — 변경은 청취 구현부만
- `flowA/A01-avatar.js`
  - `aiSpeak(text, signal)` 안에서 typewriter 와 동시에 `Audio` 객체 생성 + Spring `/api/voice/synthesize` 응답으로 재생
  - 바지인(barge-in): 사용자 발화 감지 시 `audio.pause()` + signal abort
  - typewriter 와 음성 재생의 타이밍 일치는 음성 길이 기반 typewriter 속도 조정 (옵션) 또는 단순 동시 재생

**`common/api.js` 추가**:

```js
const Voice = {
  // FormData 업로드 — 응답은 ApiResponse<{ text }>
  transcribe(audioBlob) {
    const fd = new FormData();
    fd.append('audio', audioBlob, 'speech.webm');
    return requestMultipart('POST', '/api/voice/transcribe', fd);
  },
  // 응답이 audio binary 라 별도 처리 — Blob 반환
  synthesize(text, voice) {
    return requestBinary('POST', '/api/voice/synthesize', { text, voice });
  }
};
```

`requestMultipart` / `requestBinary` 헬퍼 신설 — 기존 JSON 전용 `request()` 와 분리.

### 2-6. 비용 / 지연 추정

기준: 일 평균 100건 발화, 발화당 5초 음성 + reply 50자.

| 항목 | 단가 | 일 사용량 | 일 비용 | 월 비용 (30일) |
|---|---|---|---|---|
| Whisper STT | $0.006/분 | 100건 × 5초 = 8.3분 | $0.05 | **$1.5** |
| OpenAI TTS | $15/1M chars | 100건 × 50자 = 5,000자 | $0.075 | **$2.25** |
| **합계** | | | | **약 $3.75/월 (~5,000원)** |

지연 (가정):
- 발화 끝 감지 → blob 생성 → Spring 업로드 → Whisper → 응답: **1~2초**
- reply 텍스트 → /synthesize → Spring 응답 → Audio 재생 시작: **1~1.5초**
- typewriter 효과 1줄(50자, 50ms/자) ≈ 2.5초 — 음성 재생과 거의 일치

UX 보강:
- STT 처리 중 마이크 버튼에 spinner 또는 "듣고 있어요..." → "처리 중..." 텍스트 전환 (이미 `onConvModeChange` 의 `THINKING` 모드 활용)
- 5초 이상 응답 지연 시 토스트 "잠시만요..." (가이드 §"5초 이상 지연" 정책 유지)

### 2-7. 테스트 방향 (TDD)

순수 로직 + 모킹 가능 부분 위주 단위 테스트:

| 대상 | 테스트 종류 | 비고 |
|---|---|---|
| `OpenAiClient` 의 응답 파싱 / 에러 분기 | Spring `@WebFluxTest` 또는 `MockWebServer` | 401/429/500 에러 매핑 |
| `audio-recorder.js` 의 stop → Blob 변환 | jsdom + MediaRecorder mock | 어려움 — 통합 검증으로 대체 가능 |
| `common/api.js` 의 `requestMultipart` 응답 처리 | Node test + fetch mock | 200/4xx 분기 |
| Whisper API key 마스킹 (로그) | Node test (순수 함수) | 기존 `maskBody` 와 동일 패턴 |

### 2-8. 검증 절차

1. **로컬 단위 검증**: `npm test` + Spring 의 `@WebMvcTest` 통과
2. **통합 수동 검증**:
   - Spring 부팅 후 `curl -X POST http://localhost:8080/api/voice/transcribe -F "audio=@sample.webm"` → text 응답
   - `curl -X POST http://localhost:8080/api/voice/synthesize -d '{"text":"안녕하세요"}' -o out.mp3` → 재생 가능
3. **브라우저 end-to-end**: A01 진입 → 마이크 클릭 → 발화 → 텍스트 인식 → reply 음성 재생까지
